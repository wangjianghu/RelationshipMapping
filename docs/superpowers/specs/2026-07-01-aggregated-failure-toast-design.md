# Aggregated Failure Toast Design

## Goal

将智能推荐与智能批量映射在失败场景下的提示，从笼统的“没有成功绑定任何节点/映射”改为统计型提示，明确展示成功数、失败数和失败原因聚合结果。

## Scope

本次覆盖：

- `js/script.js`
- `关系映射.html`

本次不覆盖：

- 推荐算法
- 映射规则本身
- 导入导出逻辑
- UI 样式和布局

## Current Problem

当前外层提示文案过于笼统：

- 主页面智能推荐失败时提示 `没有成功绑定任何推荐节点`
- 主页面和旧版页面智能批量映射失败时提示 `没有成功应用任何映射`

虽然底层 `attemptBind()` 已返回 `reason`，但上层没有将失败原因转成统计型提示，导致用户无法快速知道失败原因。

## Design

采用“统一结果聚合 + 单次汇总提示”方案：

1. 保留 `attemptBind()` 的 `success/reason` 返回值
2. 新增原因码到中文文案的映射函数
3. 新增结果汇总函数，统计：
   - 成功数
   - 失败数
   - 失败原因分组计数
4. 智能推荐与智能批量映射在完成一轮操作后，仅弹一次汇总提示

## Message Format

统一使用以下格式：

- 全成功：`成功 N，失败 0`
- 全失败：`成功 0，失败 N；失败原因：原因A（X项），原因B（Y项）`
- 部分成功：`成功 N，失败 M；失败原因：原因A（X项），原因B（Y项）`

示例：

- `成功 0，失败 1；失败原因：A平台仅允许叶子节点映射（1项）`
- `成功 0，失败 3；失败原因：A平台仅允许叶子节点映射（2项），该B节点已绑定到其他A节点（1项）`
- `成功 2，失败 1；失败原因：重复绑定（1项）`

## Reason Mapping

建议使用如下映射：

- `leaf_only_a` -> `A平台仅允许叶子节点映射`
- `leaf_only_b` -> `B平台仅允许叶子节点映射`
- `b_used` -> `该B节点已绑定到其他A节点`
- `duplicate` -> `重复绑定`
- `invalid_node` -> `节点无效或不存在`
- 未识别原因 -> `未知原因`

## Toast Strategy

为了避免一轮批量操作中出现多条逐项错误提示，本次建议：

1. 为 `attemptBind()` 增加可选参数，允许调用方关闭即时错误 toast
2. 智能推荐与智能批量映射调用 `attemptBind()` 时关闭逐项 toast
3. 完成整轮处理后，由外层统一弹出一条统计型提示
4. 普通手动绑定继续保留即时错误 toast，不改变现有交互

## Affected Flows

### 主页面智能推荐

- 当前：逐项失败后，外层只显示“没有成功绑定任何推荐节点”
- 调整后：显示统计型提示，并保留成功场景下的成功计数

### 主页面智能批量映射

- 当前：逐项失败后，外层只显示“没有成功应用任何映射”
- 调整后：显示统计型提示，并聚合失败原因

### 旧版页面智能批量映射

- 当前：逐项失败后，外层只显示“没有成功应用任何映射”
- 调整后：显示统计型提示，并聚合失败原因

## Validation

需要验证以下场景，主实现与旧版页面都要一致：

1. 单条失败
   - 预期：`成功 0，失败 1；失败原因：A平台仅允许叶子节点映射（1项）`

2. 多条失败且原因相同
   - 预期：`成功 0，失败 N；失败原因：A平台仅允许叶子节点映射（N项）`

3. 多条失败且原因不同
   - 预期：`成功 0，失败 N；失败原因：...`，并正确分组计数

4. 部分成功
   - 预期：`成功 N，失败 M；失败原因：...`

5. 全成功
   - 预期：`成功 N，失败 0`

6. 普通手动绑定
   - 预期：仍保留当前即时错误提示，不受本次统计提示改动影响

## Risks

风险较低，主要有两点：

- 若不关闭 `attemptBind()` 的即时错误 toast，会出现“多条单项错误 + 一条汇总提示”的噪音体验
- 主实现与旧版页面必须共享相同的原因映射规则，否则提示口径会再次分叉
