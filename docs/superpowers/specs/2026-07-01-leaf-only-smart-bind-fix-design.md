# Leaf-Only Smart Bind Fix Design

## Goal

修复“仅允许叶子节点映射”开启后，智能推荐与智能批量映射仍可为非叶子节点创建绑定，或在绑定失败后仍显示成功提示的问题。

## Scope

本次覆盖：

- `js/script.js`
- `关系映射.html`

本次不覆盖：

- 导入导出逻辑
- 推荐算法本身
- UI 样式调整

## Current Problems

已确认存在两个缺陷：

1. 在“未映射 -> 智能批量映射 -> 应用选中映射”中，代码直接写入 `state.mappings`，绕过了叶子节点限制。
2. 在“智能推荐 -> 绑定所选”中，代码会调用底层绑定函数，但不判断是否真正绑定成功，导致提示叶子限制后仍显示“已绑定 N 个推荐节点”。

## Root Cause

普通绑定入口依赖 `attemptBind(aNode, bNode)` 处理叶子限制，但智能入口没有统一依赖“可返回结果”的底层绑定能力：

- 一个入口直接绕过底层绑定
- 另一个入口调用了底层绑定，但忽略其执行结果

## Design

采用统一绑定结果方案：

1. 将主实现和旧版页面中的 `attemptBind(aNode, bNode)` 改为返回结果对象
2. 结果对象至少包含：
   - `success: boolean`
   - `reason: string`
3. 叶子限制、B 节点占用限制、重复绑定等失败场景仍保留当前提示，但返回 `success: false`
4. 只有真正写入或更新映射时才返回 `success: true`

## Behavior Changes

### 普通绑定

- 继续走 `attemptBind()`
- 现有交互不变
- 只是新增返回值，供其他入口复用

### 智能推荐 -> 绑定所选

- 对每个目标节点调用 `attemptBind()`
- 仅在 `result.success === true` 时累计成功数
- 如果全部失败，不再显示“已绑定 N 个推荐节点”
- 失败时沿用 `attemptBind()` 内部错误提示

### 未映射 -> 智能批量映射 -> 应用选中映射

- 去掉直接 `state.mappings.push(...)` 的写法
- 改为逐条调用 `attemptBind(source, target)`
- 仅在 `result.success === true` 时累计成功数
- 成功数为 0 时，不再出现误导性的成功提示

## Result Semantics

建议统一使用以下语义：

- `success: true`
  - 新建映射成功
  - 替换映射成功
  - 追加目标成功
- `success: false`
  - `leaf_only_a`
  - `leaf_only_b`
  - `b_used`
  - `duplicate`
  - `invalid_node`

无需为本次修复引入复杂的错误聚合结构，只要保证成功计数准确即可。

## Validation

需要验证以下场景，且主实现与旧版页面都要通过：

1. 开启“仅允许叶子节点映射”，对非叶子 A 或 B 执行智能推荐“绑定所选”
   - 预期：出现叶子限制提示
   - 预期：不新增映射
   - 预期：不显示“已绑定 1 个推荐节点”

2. 开启“仅允许叶子节点映射”，在智能批量映射中应用非叶子候选
   - 预期：不新增映射
   - 预期：成功数不增加

3. 开启“仅允许叶子节点映射”，对合法叶子节点执行两种智能绑定
   - 预期：映射成功
   - 预期：成功提示数量准确

4. 关闭叶子限制后执行智能绑定
   - 预期：行为与当前合法绑定一致，不引入回归

## Risks

风险较低，主要有两点：

- `attemptBind()` 的返回值变更会影响多个调用点，需要同步检查主实现和旧版页面
- 智能入口改为依赖统一绑定结果后，成功提示文案必须与真实成功数保持一致
